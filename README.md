## 1. Системні вимоги

Для коректного функціонування програмного засобу в робочому середовищі мають бути встановлені наступні компоненти:

- **Java Runtime Environment (JRE):** Версія 17 або вище.

- **Node.js та npm:** Необхідні для роботи модуля генерації зображень вихідного коду (використовується утиліта `carbon-now` через `npx`).

- **Операційна система:** Windows (із встановленим `cmd.exe`) або UNIX-подібна система (із встановленим `zsh`).


---

## 2. Режими роботи застосунку

Застосунок підтримує два незалежні режими виконання:

1. **Базовий (дефолтний) режим:** Обробка одного Markdown-файлу (`rom.md`) для генерації єдиного звіту.

2. **Каскадний режим (`cascade`):** Пакетна обробка масиву лабораторних робіт на основі єдиної XML-конфігурації (`com.xml`).


---

## 3. Базовий режим (Default Mode)

Базовий режим призначений для трансформації конкретного об'єкта звіту (Report Object Model - ROM), описаного у форматі Markdown, у фінальний документ (DOCX та/або PDF).

### 3.1. Параметри командного рядка (CLI)

Виклик базового режиму здійснюється без вказування додаткових підкоманд.

```
java -jar report-generation.jar [опції]
```

Таблиця доступних параметрів (прапорців):

|**Прапорець**|**Повна назва**|**Опис**|**Значення за замовчуванням**|
|---|---|---|---|
|`-ctx`|`--contextPath`|Шлях до робочої директорії (контексту), в якій відбувається пошук файлів коду, зображень та збереження результату.|Поточна директорія (`user.dir`)|
|`-o`|`--outputPath`|Базова назва вихідного файлу документа (без вказування розширення).|`report`|
|`-i`|`--inputReportMd`|Абсолютний або відносний шлях до вхідного Markdown-файлу звіту. Файл обов'язково повинен містити YAML-блок (починатися з `---`).|`rom.md` у теці контексту|
|`-t`|`--template`|Шлях до власного DOCX-шаблону, який буде використано для підстановки даних.|Вбудований `/template.docx`|
|`-docx`|(відсутня)|Вказує системі на необхідність генерації вихідного файлу у форматі DOCX.|`false`|
|`-pdf`|(відсутня)|Вказує системі на необхідність генерації вихідного файлу у форматі PDF.|`false`|

_Примітка:_ Для успішного завершення роботи програми необхідно вказати хоча б один формат виводу (`-docx` або `-pdf`).

### 3.2. Структура вхідного файлу (ROM - Report Object Model)

Вхідний файл (стандартна назва `rom.md`) повинен складатися з двох частин:

1. **YAML Frontmatter:** Блок метаданих, що описує конфігурацію звіту (обмежений тегами `---`).

2. **Markdown Content:** Тіло звіту (хід роботи), що містить текст та посилання на впроваджені зображення у форматі `{{назва_фото}}`.


#### Специфікація YAML Frontmatter:

- `properties`: Словник (Key-Value) для визначення скалярних змінних, які будуть підставлені у шаблон (наприклад: назва курсу, ПІБ студента). Рядки рекомендується брати в одинарні лапки.

- `codes`: Масив файлових шляхів (у форматі ANT glob-паттернів, наприклад `*/**/**.java`). Система автоматично знайде ці файли у контекстній директорії та додасть їхній вміст у розділ "Код програми" (Додатки).

- `photos`: Ієрархічний блок для генерації та впровадження зображень. Поділяється на три підкатегорії:

    - `bash`: Виконання команд у терміналі із подальшим фотографуванням результату.

    - `text`: Фотографування вмісту конкретного текстового файлу.

    - `images`: Використання існуючого файлу зображення з файлової системи.


#### Приклад `rom.md`:

```
---
properties:
  course_name: 'Аналіз даних'
  lab_name: 'Реалізація алгоритмів'
  lab_number: 1
  student_name: 'Петренко П.П.'
codes:
- '*/**/**.java'
photos:
  bash:
    terminal_run_1:
      label: 'Результат виконання програми'
      slice: 'end-20..'
      runs:
        - command: 'java Main'
          input: '4\n5\n'
  text:
    source_code_photo:
      name: 'Main.java'
      label: 'Вихідний код класу Main'
      slice: '10..50'
  images:
    diagram:
      file: 'assets/diagram.png'
      label: 'Архітектурна діаграма'
---
## Хід роботи
У цій лабораторній роботі було розроблено алгоритм.
{{diagram}}
Тестування алгоритму наведено нижче:
{{terminal_run_1}}
{{source_code_photo}}
```

#### Параметри конфігурації фотографій:

- `label`: Текстовий підпис, що буде розміщений під зображенням у звіті (формат "Рис. N [label]"). Доступно для всіх типів.

- `slice`: Діапазон рядків, який буде залишено перед фотографуванням. Підтримується синтаксис "від..до" (наприклад, `5..10`), а також ключове слово `end` (наприклад, `end-20..`). Доступно для `bash` та `text`.

- `runs`: Масив об'єктів для категорії `bash`. Кожен об'єкт містить `command` (команду для термінала) та опціональний `input` (дані для імітації користувацького вводу stdin).

- `name`: Назва файлу для категорії `text` (пошук здійснюється рекурсивно у контекстній директорії).

- `file`: Точний відносний шлях до файлу для категорії `images`.

## 4. Каскадний режим (Cascade Mode)

Каскадний режим забезпечує масову ініціалізацію та генерацію серії лабораторних робіт на основі єдиної специфікації (Course Object Model - COM), написаної у форматі XML.

### 4.1. Параметри командного рядка (CLI)

```
java -jar report-generation.jar cascade [опції]
```

Таблиця доступних параметрів (прапорців):

|**Прапорець**|**Повна назва**|**Опис**|**Значення за замовчуванням**|
|---|---|---|---|
|`-com`|`--CourseObjectModel`|Шлях до файлу конфігурації курсу у форматі XML.|`com.xml` у поточній директорії|
|`-b`|`--build`|Вказує системі на необхідність розпочати фазу компіляції звітів (DOCX/PDF) одразу після фази створення файлової структури.|`false` (тільки генерація структури)|
|`-same`|`--sameDirectory`|Інструкція зберігати та генерувати об'єкти всіх лабораторних робіт в одній (кореневій) директорії, уникаючи створення підкаталогів `lab-1`, `lab-2` тощо.|`false`|

_Примітка:_ Якщо файл `com.xml` не знайдено за вказаним шляхом, система автоматично згенерує еталонний файл `com.xml` (разом з XSD-схемою) у поточній директорії та завершить роботу.

### 4.2. Життєвий цикл каскадної обробки

Виконання каскадного режиму складається з наступних етапів:

1. **Десеріалізація:** Читання та валідація файлу `com.xml` (через JAXB та Hibernate Validator).

2. **Розгортання робочого середовища (послідовно для кожної `<lab>`):**

    - Створення робочої директорії (залежно від прапорця `-same`).

    - Виконання підготовчих shell-команд (тег `<runs>`). Ці команди не потрапляють у звіт, вони слугують для налаштування середовища (наприклад, `npm init` або створення `virtualenv`).

    - Створення заданих у специфікації файлів (тег `<files>`).

    - Створення файлу моделі звіту (наприклад, `rom.md`) на основі вмісту тегу `<report>`.

3. **Генерація звітів (якщо вказано прапорець `-b`):**

    - Паралельний (через `ExecutorService`) або послідовний запуск базового алгоритму побудови DOCX та PDF для кожної ініціалізованої лабораторної роботи.


### 4.3. Структура файлу конфігурації (COM - Course Object Model)

Кореневий тег `<course>` містить масив елементів `<lab>`.

Кожен елемент `<lab>` складається з наступних блоків:

- `<runs>`: Блок ініціалізаційних команд оболонки.

    - Містить елементи `<run command="..." input="..." />`.

- `<files>`: Блок файлів, що мають бути створені у робочій директорії.

    - Містить елементи `<file name="відносний/шлях.ext">...вміст...</file>`. Увесь вміст повинен бути загорнутий у секцію `<![CDATA[ ... ]]>`.

- `<report>`: Повний вміст файлу `rom.md` (включно з YAML Frontmatter). Увесь текст також має бути загорнутий у секцію `<![CDATA[ ... ]]>`.


#### Фрагмент прикладу `com.xml`:

```
<course xmlns="com.banew/report-generation/course-object-model">
    <labs>
        <lab>
            <runs>
                <run command="npm init --yes" />
            </runs>
            <report><![CDATA[
---
properties:
  lab_number: 1
photos:
  bash:
    test_run:
      label: 'Тестування'
      runs:
        - command: 'npm start'
---
Текст звіту.
{{test_run}}
            ]]></report>
            <files>
                <file name="src/index.js"><![CDATA[
console.log("Hello Enterprise");
                ]]></file>
            </files>
        </lab>
    </labs>
</course>
```

## 5. Обробка шаблонів (Templating Engine)

Система використовує рушій шаблонів **Velocity** спільно з **XDocReport** для підстановки даних.

**Вимоги до еталонного документа (DOCX-шаблону):**

- Для підстановки текстових властивостей (з блоку `properties` файлу `rom.md`) використовуються маркери виду `${property_name}`.

- Для впровадження тіла звіту (включаючи форматований HTML, згенерований з Markdown) використовується спеціальний маркер `${content}`. Програмний засіб виконує низькорівневу модифікацію XML-структури DOCX (перетворення на `MERGEFIELD`), щоб забезпечити підтримку складного форматування (Rich Text).

- Для підстановки масиву кодів використовується синтаксис Velocity безпосередньо в тілі DOCX-шаблону:

    ```
      #if(!$codeMap.isEmpty())
        Код програми
        #foreach($entry in $codeMap.entrySet())
          ${entry.key}
          ${entry.value}
        #end
      #end
    ```

## 6. Завершення роботи та коди статусу

Система дотримується стандартних угод про коди виходу операційної системи:

- `0`: Успішне виконання всіх завдань.

- `> 0`: Наявність критичної помилки (відсутність файлів, синтаксична помилка YAML/XML, помилка генерації зображень, ненульовий код виходу в оболонці термінала). Деталі помилки фіксуються в журналах виконання (log).
