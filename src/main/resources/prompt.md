Ось, що відбудеться після того, як я його одержу:
1. Для УСІХ лабораторних робіт буде створено окрему теку, в якій будуть викликані наступні дії:
   1. Буде викликано усі команди з <runs>
   2. Будуть створені всі файли з <files>
   3. Буде створено опис звіту "rom.md" з тегу <report>
2. На основі опису звіту "rom.md" відбудеться побудова фінального DOCX/PDF звіту, зокрема:
   1. Будуть викликані та побудовані фото для звіту в порядку "bash" -> "text" -> "images" (тобто, до прикладу, можна використати такі "images", які створюються при виклику команд в "bash" і тд)
   2. Залежно від наявних даних побудується звіт приблизно у такому порядку (з використанням властивостей з YAML frontmatter):
      1. Титульна сторінка (там всі ті назви, ПІБ і тд)
      2. Мета роботи ("lab_purpose")
      3. Хід роботи (ЦЕ ЯКРАЗ ТІЛО MD ФАЙЛУ - тобто все те, що йде після YAML frontmatter разом з вже вкладеними фото)
      4. Висновок ("lab_conclusion")
      5. Код програми (всі файли, задані паттернами з "codes")

Як результат: залежно від завдання конкретної лабораторної роботи, постарайся досконало її використати,
використовуючи можливості даного XML та, зокрема, вкладеного в нього MD файлу.

Додаткові настанови:
- не пиши явно підписи до фото в ході роботи саме в MD; для цього існують лише і тільки "label"
- ніколи не допускай великих по вмісту фото, завжди по можливості обрізай через "slice"
- у всіх YAML frontmatter частинах огортай всі СВОЇ рядки в лапки, аби не було проблем, до
  прикладу не course_name: Аналіз даних, а course_name: "Аналіз даних"
- чітко розрізняй блок <runs> в загальному XML (який виконується з самого початку ініціалізації теки для лаби і має 
  всього лише підготувати теку для роботи програми, яку передбачається написати в лабі, як-от створити virtualenv для
  лаби на python, виконати npm init при лабі на JS тощо) та різновид фото "bash" в MD (який відпрацьовує САМЕ для звіту,
  тобто не тільки для фото термінала, а й для будь-яких інших дій). До прикладу, в разі, якщо потрібно по лабі, скажімо,
  якась візуалізація, то просто додатково зберігай її у файл при через створення зображення "bash" (яке, зокрема, може й
  не використовуватись), а далі просто розташовуй файл у звіті через "images"
- якщо по ходу завдання лаби треба виконати якусь візуалізацію або загалом написати БУДЬ-ЯКУ віконну програму, то
  попри те, що тоді ймовірно ти напряму зберігатимеш в коді це у файл, все ще ВІДКРИВАЙ її (тобто викликай plt.show() 
  і тд), адже всі написані тобою програми мають бути репрезентабельними І БЕЗ ЗВІТУ. При автоматизованій побудові звіту 
  ці вікна автоматично закриватимуться, а ось якщо відповідний код запустить людина, то їй правильно покажеться вікно.
- пишучи нижню MD-частину звітів (та, що "хід роботи"), по максимуму використовуй можливості Markdown, як-от виділення
  заголовків "##" і тд
- ЩЕ РАЗ: при генерації тек ПЕРШЕ виконуються команди з <runs>, а ПОТІМ вже створюються файли. Ти НЕ МОЖЕШ
  використовувати файли з <files> в командах <runs>!!! Щоб використовувати створені файли використовуй MD звіти,
  а саме їх різновид фото "bash"!
- не використовуй кодові фрагменти `````` в MD-звітах
